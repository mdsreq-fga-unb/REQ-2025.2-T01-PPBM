## Atividades e Técnicas de ER

### Elicitação e Descoberta

**Entrevistas com stakeholders:** entrevistas realizadas com a stakeholder responsável pela idealização do PPBM e uma das professoras do projeto, onde nos ajuda a entender desejos e necessidades do Programa Bombeiro Mirim, com ideias capazes de resolver o problema enfretado.

**Brainstorming:** Sessões realizadas entre a equipe e as vezes com a cliente permitiram gerar propostas de soluções e ideias para atender às necessidades do projeto, estimulando a livre expressão de ideias e criatividade.

### Análise e consenso

**Priorização MoSCoW:** A priorização MoSCoW é uma técnica utilizada para classificar requisitos ou funcionalidades com base em sua importância para o sucesso do projeto. Representam as categorias "Must have", "Should have", "Could have" e "Won't have". Muito útil para definir quais funcionalidades teriam prioridade no desenvolvimento, considerando impacto no negócio e esforço necessário. 

**Priorização ICE:** É uma técnica para classificar e priorizar iniciativas, tarefas ou projetos com base em três critérios principais: Impacto, Confiança e Esforço. Na nossa equipe, ela serve para gerar uma pontuação de prioridade antes do sprint planning.

### Declaração de Requisitos

**User Stories**: São representações de requisitos que descrevem o que o sistema deve fazer para entregar valor e como o usuário interage com ele.

### Representação de Requisitos

***Prototipação de Interface:*** É uma técnica utilizada para criar protótipos de interfaces gráficas para validar ideias e prototipar soluções. Muito usado para testar ideias e validar a viabilidade de uma solução, além de ajudar a stakeholder a visualizar a solução e validar.

### Verificação e Validação de Requisitos

**Feedback do Cliente:** Uma opinião ou comentário dado pelo cliente através do que foi entregue. É uma forma de verificar se atende às necessidades do usuário e orienta futuras melhorias.

**Definition of Done (DoD):** DoD é uma lista clara e objetiva de tudo que precisa ser feito para uma tarefa ou história ser considerada 100% pronto. Aqui, ela terá o objetivo de complementar os critérios de aceitação e de abranger todo o requisito e sua qualidade.

**Definition of Ready(DoR):** DoR é uma lista ou conjunto de critério de tudo que a história ou tarefa precisa cumprir antes de ser puxada para o desenvolvimento. a DoR deve ser cumprida durante a sprint planning para que a DoD seja executada na nossa equipe.

### Organização e Atualização de Requisitos

**Controle de Versão**: Ao utilizar o modelo Kanban, a equipe consegue se organizar e manter controle de versões dos requisitos, garantindo que os requisitos não sejam alterados sem aviso prévio.

**Priorização contínua:** Revisão das prioridades usando MoSCoW e a matriz de valor e complexidade. 

### Ferramentas Utilizadas

#### Comunicação:
**Google Meet:** Ferramenta utilizada para reuniões com stakeholder.

**Discord:** Ferramenta utilizada para comunicação entre membros da equipe e dailys.

#### Gestão e Controle:

**Github Projects:** Backlog e gestão de tarefas.

**GitHub Issues:** Criação de issues para cada história de usuário e acompanhamento detalhado do projeto.

**Git:** Controle de versão de código e documentação.
